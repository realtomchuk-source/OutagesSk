# DATA_README – Структура даних та життєвий цикл інформації

Цей документ описує формати файлів, логіку обробки даних та роботу адмінки.

## 1. Огляд файлів даних

У папці `data/`:
- `villages.json` – перелік населених пунктів
- `districts.json` – округи з прив'язкою до сіл
- `outages_snapshot.json` – актуальний стан відключень після фільтрації
- `messages.json` – база повідомлень (стрічка + Telegram)
- `outages_previous.json` – попередній знімок (у планах для оптимізації GitHub Actions)
- `update_log.json` – журнал перевірок (у планах)

## 2. Довідники `villages.json` та `districts.json`

- `villages.json`: масив назв населених пунктів без скорочень «с.» чи «м.». Фільтрація за входженням підрядка.
- `districts.json`: словник, де ключ – назва округу, значення – масив сіл округу. Місто Старокостянтинів обробляється окремо.

## 3. `outages_snapshot.json` (результат збору)

Формат – масив об’єктів:
{
  "settlement": "м. Старокостянтинів",
  "type": "Аварійні",
  "created_date": "10.05.2026",
  "start_datetime": "2026-05-10T09:35",
  "end_datetime": "2026-05-10T23:50",
  "streets": ["вул. Шевченка", "вул. Замкова"],
  "streets_detailed": [
    {"name": "вул. Шевченка", "houses": "1, 2, 4, 10"},
    {"name": "вул. Замкова", "houses": ""}
  ]
}

Поля:
- `settlement` – оригінальний рядок з адреси
- `type` – «Аварійні» або «Планові»
- `start_datetime`, `end_datetime` – ISO-формат для порівняння
- `streets` – масив назв вулиць без номерів будинків
- `streets_detailed` – масив об'єктів із деталізацією номерів будинків (для вкладки "Сирі дані")

## 4. `messages.json` – база повідомлень

Це центральне сховище результатів форматувальника. Містить масив об’єктів:

Приклад запису для стрічки:
{
  "id": "2026-05-12_feed",
  "date": "2026-05-12",
  "type": "feed",
  "target": null,
  "content": "<h3>⚡ Оперативна інформація...</h3>",
  "created_at": "2026-05-12T06:10:00"
}

Приклад запису для Telegram-посту на 13 травня:
{
  "id": "2026-05-13_tg",
  "date": "2026-05-13",
  "type": "telegram_post",
  "target": null,
  "content": "🏙️ м. Старокостянтинів\n🟡 Планові...",
  "created_at": "2026-05-12T06:10:00"
}

Поля:
- `id` – унікальний ключ (дата_тип). Гарантує перезапис (Upsert) при повторній генерації за ту саму дату без дублювання.
- `date` – дата, на яку призначено повідомлення (для стрічки – сьогодні, для Telegram – конкретний день).
- `type`: `"feed"` або `"telegram_post"`.
- `content` – згенерований текст (HTML або звичайний текст).
- `created_at` – час генерації (ISO 8601). Використовується для очищення старих записів.

**Важливо:**
- Стрічка (`feed`) генерується на **сьогоднішній день**, але містить дані на сьогодні та завтра.
- Telegram-пости генеруються окремо на **кожен день** (від сьогодні до +4 дні), лише якщо на цей день є активні відключення. Якщо пост для дати вже існує, він перезаписується свіжим варіантом.
- Якщо для дати раніше був згенерований пост, а тепер відключення зникли, пост оновлюється текстом «🔕 На цей день відключення не заплановані.».

## 5. Логіка визначення активності відключення

Відключення вважається **активним** на конкретну дату, якщо його часовий інтервал перетинається із цією добою:
- Початок доби: 00:00
- Кінець доби: 23:59
Умова: `start_datetime <= кінець_доби AND end_datetime >= початок_доби`.

Це дозволяє коректно обробляти тривалі (багатоденні) відключення.

## 6. Очищення бази

Очищення виконує `formatter.py` після додавання нових повідомлень. Видаляються всі записи з `messages.json`, у яких `created_at` старіше 7 днів.
Скрипт читає поточний масив, додає або оновлює нові повідомлення за `id`, відфільтровує старі та перезаписує файл.

## 7. Адмінка (`admin.html`)

**Розташування:** корінь репозиторію. Статична сторінка з JavaScript.
**Автентифікація:** пароль `ADMIN_PASSWORD` зберігається в GitHub Secrets. Під час генерації `admin.html` у неї вбудовується SHA-256 хеш цього пароля. Користувач вводить пароль, JS обчислює хеш і порівнює.

**Відображення:**
- **Вкладка «Стрічка»:** показує дві колонки – на сьогодні та на завтра (дані з останнього запису `type: "feed"`).
- **Вкладка «Telegram-пости»:** список постів, відсортований за датою. Кожен пост – картка з датою та текстом.
- **Архів:** можливість вибрати будь-яку дату в межах наявних даних і переглянути відповідний пост чи стрічку.
- **Сирі дані:** табличне відображення даних з `outages_snapshot.json` (без обробки ШІ) з можливістю розгортання списку номерів будинків для кожної вулиці.

**Редагування:** адмінка працює лише для перегляду. Жодних змін через інтерфейс не передбачено.

## 8. Логування (`update_log.json`) (В процесі розробки)

Формат:
[
  {
    "timestamp": "2026-05-11T08:00:00",
    "new_data_found": true,
    "records_count": 5,
    "status": "ok"
  },
  ...
]
- `new_data_found` – чи змінились дані з попереднього знімка.
- `status` – `ok`, `no_change`, `html_structure_error`, `http_error`.

## 9. Валідатор виходу LLM

Вбудований у `formatter.py`. Перевіряє:
- наявність усіх переданих вулиць у згенерованому тексті;
- відсутність вигаданих назв.

При помилці (відсутності вулиці) скрипт автоматично робить до 2 повторних спроб генерації. Якщо помилка залишається, до тексту додається автоматичне попередження `⚠️ УВАГА: Можливо, ШІ переніс не всі вулиці. Перевірте джерело.`.
